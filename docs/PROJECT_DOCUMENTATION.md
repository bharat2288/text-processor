## 1. Background and Motivation

This project emerged from a concrete problem during dissertation writing at UC Berkeley. The dissertation examined Crypto Twitter as a site of informal learning, tracing how discourse genres, visual devices, and algorithmic ranking produce particular kinds of speculators. It drew on a diverse interdisciplinary literature: Deleuze, Bourdieu, conversation analysis, platform studies, science and technology studies, economic anthropology, and learning sciences.

The challenge was not just reading these texts but *retrieving precise passages* during the writing process. With 300+ sources spanning theory, ethnography, and methodology, the question was: how do you query your literature programmatically and get back faithful, contextualized responses?

In 2024, when this project began, LLM hallucination rates were substantially higher than they are now. Asking ChatGPT or Claude to summarize or cite an academic text would frequently produce fabricated quotes, invented page numbers, and confident misattributions. For academic work where fidelity to source material is non-negotiable, the standard "just ask the AI" approach was not viable.

---

## 2. The Problem Being Solved

**Core requirement:** Query dissertation literature and receive answers that are *deterministically grounded* in the actual source texts — no hallucination, no paraphrase drift, no invented citations.

**Why existing tools failed:**
- **ChatGPT/Claude (2024):** Hallucinated citations, fabricated quotes, couldn't reliably distinguish between what a text said vs. plausible-sounding content
- **PDF readers with AI features:** Worked for single documents but couldn't handle cross-text queries across hundreds of sources
- **Academic reference managers (Zotero, Mendeley):** Good for bibliographic management but no semantic search or AI-assisted retrieval
- **Manual search:** Ctrl+F across 315 PDFs was possible but prohibitively slow for the kind of thematic, cross-cutting queries needed during writing

**The solution:** Convert all PDFs into semantically chunked text, upload to OpenAI vector stores, enrich with structured Q&A pairs, and query via CustomGPTs that were constrained to only retrieve from the vector store (no generative hallucination). This created a deterministic retrieval system: every answer was traceable to a specific chunk from a specific text.

---

## 3. System Overview

The system evolved through two major versions:

### Version 1.0: Complex Pipeline (Jan 2024 – Nov 2025)
Full cloud integration with OpenAI. PDFs were chunked locally, uploaded to OpenAI's file storage, added to a vector store, enriched with Q&A pairs generated by custom OpenAI Assistants, and all metadata tracked in Excel and JSON state files.

### Version 2.0: Simple Pipeline (Nov 2025 – Jan 2026)
Local-only processing. Dropped the OpenAI upload and Q&A enrichment steps entirely. Just semantic chunking and text output. 6x faster, zero API cost, no encoding issues.

Both versions shared the same output directory structure and could coexist — the simple pipeline wrote to the same `chunked_txts_concatenated/` and `chunked_txts_by_author/` directories that the complex pipeline created.

## 4. Chronological Development

### Phase 1: Early Scripts and Manual Processing (Early 2024)

The project began with standalone scripts for individual tasks:

- **`file_chunker_semantic.py`** was the first script written. It used PyMuPDF to extract text from PDFs, spaCy for sentence segmentation, and tiktoken for token counting. It produced JSON files with semantically coherent chunks of ~600 tokens each, with 20% overlap between adjacent chunks to preserve context at boundaries.

- **`streamlined-chunks-processor.py`** (now archived) handled the conversion from JSON chunks to plain text formats suitable for Claude's context window.

- Manual metadata entry tracked what had been processed via a simple Excel spreadsheet.

At this stage, the workflow was: run the chunker on a PDF, manually upload the resulting JSON to OpenAI, manually create the Q&A enrichment, manually track what had been done. Each PDF required multiple manual steps.

### Phase 2: OpenAI Integration and QA Enrichment (Mid 2024)

The key insight driving this phase was that raw text chunks, while useful for retrieval, didn't capture the *conceptual structure* of the texts. Two separate OpenAI Assistants were created:

1. **Generator Assistant (`AUTHOR_QA_GEN_V2_ID`):** Took a chunked text file and generated structured question-answer pairs that captured the key arguments, concepts, and claims in each chunk. The questions were phrased as the kinds of queries a researcher might make: "What does Bourdieu mean by habitus in this section?", "How does Zaloom characterize the relationship between affect and prediction?"

2. **QC Assistant (`AUTHOR_QA_QC_V2_ID`):** Quality-controlled the generated Q&A pairs, checking for accuracy against the source text, flagging hallucinated content, and rejecting outputs that didn't meet quality standards.

**`revised_runqav2.py`** automated this two-stage process:
1. List all files in the vector store
2. Identify which ones haven't been processed yet (using `processed.json` as state)
3. For each unprocessed file, create a thread with the Generator Assistant
4. Send the Generator's output to the QC Assistant for validation
5. If QC passed: save the Q&A JSON locally, upload it to the vector store alongside the source text, update state files
6. If QC flagged issues: skip the file and log the failure

This was the most expensive and fragile part of the system. Each Q&A generation took 2-3 minutes of API time, was subject to rate limits, and occasionally produced outputs the QC rejected.

### Phase 3: Integrated Pipeline v0 (Late 2024)

**`integrated-pipeline-v0.py`** (now in `archived/`) combined all the steps into a single script. Running it on a PDF would:

1. Chunk the PDF into semantic JSON segments
2. Upload the JSON to OpenAI's file storage
3. Add it to the vector store
4. Run QA generation and QC
5. Convert the JSON to Claude-compatible text formats
6. Prompt the user for metadata (author, year, title, text type, tags)
7. Update the metadata Excel

The directory structure at this point was nested:
```
Literature/
├── chatgpt/
│   ├── chunked_jsons_concatenated/
│   └── qa_jsons/
├── claude/
│   ├── chunked_txts_concatenated/
│   └── chunked_txts_by_author/
└── pdfs/
```

### Phase 4: Centralized Configuration and Flattened Structure (Jan 2025)

After encountering repeated path issues and hardcoded paths scattered across scripts, the project was refactored:

- **`config.py`** became the single source of truth for all paths, API keys, and processing parameters. Every script imported from it rather than defining paths inline.

- The directory structure was flattened. Instead of nested `chatgpt/` and `claude/` folders, everything moved under a single `data/` directory:
```
pdf_processing/
├── data/
│   ├── pdfs/
│   ├── chunked_jsons_concatenated/
│   ├── qa_jsons/
│   ├── chunked_txts_concatenated/
│   └── chunked_txts_by_author/
└── scripts/
```

- **`update_scripts_to_config.py`** automated the migration of all scripts from hardcoded paths to `config.py` imports.

- **`lit-migrate-directory-structure.py`** moved files from the old nested structure to the new flat one.

- **`fix-complete-text-filenames.py`** corrected naming inconsistencies in the already-processed Claude text files (some had duplicated author names, inconsistent separators, etc.).

### Phase 5: Enhanced Pipeline v1 with Error Diagnostics (Jan–Aug 2025)

**`integrated-pipeline.py`** (the final version of the complex pipeline) added comprehensive error handling:

- Detailed diagnostic output when QA generation failed (stdout/stderr capture from subprocess calls)
- Pattern-based error guidance ("Rate limit" → wait 60 seconds; "Authentication" → check API key; "assistant" → check Assistant IDs)
- Rollback capability: if any step failed after uploading to OpenAI, the uploaded files were automatically deleted
- 10-minute timeout for QA generation (previously implicit)
- Graceful handling of QA generation that "succeeded with warnings" (script exited non-zero but produced the file ID)
- Better handling of keyboard interrupts during interactive metadata entry

**`reverse-integrated-pipeline.py`** was created as the inverse operation — given a PDF, it would find all related files (JSON chunks, QA JSONs, Claude texts, author folders), remove entries from the OpenAI vector store, delete local files (with optional backup), and update the metadata Excel. This was essential for cleanup when processing went wrong.

### Phase 6: Management and Maintenance Tools (Feb–Aug 2025)

As the vector store grew to hundreds of files, management tools became necessary:

- **`vector-store-manager.py`** provided CLI access to query, search, list, export, and remove files from the OpenAI vector store. Batch operations (batch-remove, batch-remove-search) were added after several instances of needing to clean up test files or re-process groups of texts.

- **`update_vector_metadata.py`** allowed updating metadata attributes on files already in the vector store (author corrections, tag additions, etc.) without re-processing the entire pipeline.

- **`check-directory-files.py`** compared file counts across all pipeline directories to detect misalignments (PDFs without corresponding chunks, orphaned QA files, etc.).

- **`standardize_lit_dirs.py`** normalized the author folder naming in `chunked_txts_by_author/` — different pipeline runs had produced slightly different folder name formats.

- **`directory_mapper_condensed.py`** and **`directory_mapper_full.py`** generated maps of the entire directory structure for reference and debugging.

- **`list_filenames.py`** was a utility for generating formatted file lists (comma-separated, Python lists, newlines) for use in scripts or reports.

- **`vector-store-web-ui.py`** (archived) was a Streamlit web interface for the vector store manager, providing a visual dashboard for browsing and managing files. It was ultimately unnecessary — the CLI was sufficient.

### Phase 7: Simple Pipeline and Sunset (Nov 2025 – Jan 2026)

By late 2025, two things had changed:

1. **LLM capabilities improved dramatically.** Claude and ChatGPT could now handle much larger context windows and were far less prone to hallucination. The elaborate Q&A enrichment layer that had been essential in 2024 was now mostly redundant — you could upload a complete text and get reliable responses.

2. **The dissertation was written.** The primary use case (deterministic literature retrieval during writing) had been fulfilled.

**`simple_pipeline.py`** (in `scripts/simple-pipeline/`) was written as a clean-slate replacement:
- Local-only processing, no OpenAI API calls
- PDF → semantic chunks → text files in ~30 seconds (vs. ~3 minutes with the complex pipeline)
- Same output format and directory structure as the complex pipeline
- No JSON intermediate format, no Q&A enrichment, no state files beyond a simple metadata Excel
- Self-contained with its own `config_simple.py` and `test_pipeline.py`

A **`MIGRATION_GUIDE.md`** documented the transition from the complex to simple pipeline.

The project was effectively archived in January 2026, with all 315 PDFs processed and 66 additional "Cited Books" stored separately.

## 5. Pipeline Architecture (v1.0 — Complex)

### Data Flow

```
PDF file (e.g., "Graeber, D. (2018) - Bullshit Jobs.pdf")
    │
    ▼
[file_chunker_semantic.py]
    │  - PyMuPDF extracts raw text page by page
    │  - spaCy segments into sentences
    │  - tiktoken counts tokens
    │  - Sentences accumulated into ~600-token chunks with 20% overlap
    │  - Minimum chunk size: 80 tokens (prevents fragment chunks)
    │  - Each chunk tagged with: chunk_id, section heading, page_start
    │
    ▼
JSON chunk file (in chunked_jsons_concatenated/)
    │  - Array of chunk objects: {chunk_id, text, section, page_start, author, year, title, source}
    │
    ▼
[Upload to OpenAI Files API]
    │  - POST to /v1/files with purpose="assistants"
    │  - Returns file_id (e.g., "file-BKnmVyVdLVpkWJNR36etfw")
    │
    ▼
[Add to Vector Store]
    │  - POST to /v1/vector_stores/{TEXT_STORE_ID}/files
    │  - File is now searchable via vector similarity
    │
    ▼
[revised_runqav2.py — QA Generation]
    │  - Creates OpenAI Thread
    │  - Sends to Generator Assistant (AUTHOR_QA_GEN_V2_ID)
    │  - Generator produces structured Q&A pairs from the chunked text
    │  - Output sent to QC Assistant (AUTHOR_QA_QC_V2_ID) for validation
    │  - If QC passes: save QAv2 JSON locally, upload to vector store
    │  - If QC fails: skip, log warning
    │
    ▼
QAv2 JSON file (in qa_jsons/)
    │  - Named: QAv2_{sanitized_name}_{file_id}.json
    │  - Also uploaded to vector store alongside source text
    │
    ▼
[process_json_to_claude_formats()]
    │  - Reads JSON chunks
    │  - Creates two output formats:
    │
    ├──▶ Complete text file (in chunked_txts_concatenated/)
    │      - Single file: LIT-{Author}_{Year}_{Title}_COMPLETE.txt
    │      - Header block with metadata
    │      - All chunks concatenated with "======= CHUNK 001 =======" markers
    │
    └──▶ Individual chunk files (in chunked_txts_by_author/)
           - Folder: {Author}_{Year}_{TitleWords}_{ChunkCount}chunks/
           - Files: LIT-{Author}-{Title}-chunk_001.txt through chunk_NNN.txt
           - One chunk per file for granular retrieval
    │
    ▼
[Interactive metadata entry]
    │  - User confirms/edits: author, year, title
    │  - Selects text type (Book, Journal Article, etc.)
    │  - Enters tags (capitalism, ethnography, crypto, etc.)
    │
    ▼
[Metadata updates]
    - pdf_metadata.xlsx: new rows for chunk file and QAv2 file
    - processed.json: marks file_id as processed (prevents re-processing)
    - meta_index.json: links text file to its QAv2 counterpart
    - Vector store attributes updated with metadata fields
```

### OpenAI Vector Store Architecture

The vector store (identified by `TEXT_STORE_ID`) contained two types of files per source text:

1. **Chunked text (source_type: "chunked"):** The raw semantic chunks from the PDF, searchable by vector similarity. When a CustomGPT queried "What does Bourdieu say about habitus?", it would retrieve the most relevant chunks across all texts.

2. **QAv2 enrichment (source_type: "qav2"):** Structured question-answer pairs generated from the chunks. These provided an additional retrieval layer — the CustomGPT could match on both the raw text and the anticipated questions a researcher might ask.

Each file carried metadata attributes: `author`, `year`, `title`, `text_type`, `tags`, `source_type`, and (for QAv2 files) `source_chunk_id` linking back to the parent text.

### CustomGPT Integration

The vector store was attached to CustomGPTs configured with instructions to:
- Only answer from retrieved content (no generative responses)
- Always cite the source text by author, year, and chunk reference
- Flag when retrieved content didn't directly answer the query rather than fabricating an answer
- Present multiple relevant passages when a query matched across texts

This created a deterministic retrieval system: every response was traceable to specific chunks, and the Q&A enrichment improved recall by anticipating research-oriented queries.

    │
    ▼

    │  - Direct output to text files

             chunk_count, processed_date, complete_file, author_folder

## 8. Key Technical Decisions

### Semantic Chunking Strategy

**Decision:** 600-token chunks with 20% overlap, minimum 80 tokens, sentence-boundary-aware.

**Rationale:** The 600-token limit was chosen to balance retrieval precision (smaller chunks = more precise matches) against semantic coherence (larger chunks = more context). The 20% overlap ensured that ideas spanning chunk boundaries weren't lost. The 80-token minimum prevented degenerate micro-chunks from section headers or sparse pages. Sentence-boundary awareness (via spaCy) prevented mid-sentence splits that would create incoherent chunks.

**Trade-offs:** Smaller chunks (e.g., 300 tokens) would have improved retrieval precision but required more chunks per text, increasing vector store costs and query latency. Larger chunks (e.g., 1200 tokens) would have preserved more context but reduced retrieval precision for specific claims or passages.

### Two-Stage QA Generation (Generator + QC)

**Decision:** Separate Generator and QC Assistants rather than a single combined Assistant.

**Rationale:** A single Assistant tended to both generate *and* approve its own Q&A, which didn't catch hallucinations. The two-stage approach forced the QC Assistant to independently evaluate the Generator's output against the source material, rejecting Q&A pairs that couldn't be grounded in the text.

**Trade-offs:** Doubled the API cost and processing time per file. Approximately 10-15% of outputs were rejected by QC, requiring re-processing or manual review.

### Vector Store vs. Embedding Database

**Decision:** OpenAI's hosted vector store rather than a local embedding database (Chroma, Pinecone, etc.).

**Rationale:** The primary consumer was CustomGPTs, which had native integration with OpenAI vector stores. A local database would have required building a separate retrieval API and connecting it to ChatGPT through a custom action — significantly more infrastructure.

**Trade-offs:** Vendor lock-in, recurring storage costs, and no control over the embedding model or similarity algorithm. But for the dissertation timeline, speed of deployment was more important than architectural purity.

### Dual Output Formats (Complete + By-Author Chunks)

**Decision:** Generate both a single concatenated text file and individual chunk files per text.

**Rationale:** Different use cases required different formats. Complete texts were uploaded to Claude Projects for full-context analysis ("Summarize the argument of this entire book"). Individual chunks were used for targeted retrieval and cross-text thematic analysis.

## 10. Lessons Learned

### On Building AI-Assisted Research Tools

**The Q&A enrichment layer was the right idea at the wrong time.** In 2024, when LLMs were more prone to hallucination, having pre-generated, QC-validated Q&A pairs significantly improved retrieval quality. By late 2025, improvements in base model capability made this layer mostly redundant. The raw chunks were sufficient for reliable retrieval.

**Semantic chunking > naive splitting.** The investment in spaCy sentence segmentation, overlap, and minimum chunk sizes paid off. Naive page-based or fixed-character splitting produced chunks that broke mid-sentence, mixed unrelated sections, and gave poor retrieval results. The extra 10 seconds of processing per PDF (for NLP-based chunking) was worth it.

**Naming conventions matter enormously at scale.** The single most time-consuming maintenance task was fixing filename inconsistencies. Different pipeline versions, different PDF naming conventions from different sources (library downloads vs. publisher PDFs vs. scraped web content), and filesystem limitations (Windows path length, special characters) all created drift. The `fix-complete-text-filenames.py` and `standardize_lit_dirs.py` scripts were both written reactively after problems accumulated.

**State management is harder than processing.** The pipeline itself was straightforward (chunk, upload, generate, save). What was genuinely complex was tracking *what had been processed*, *what needed re-processing*, and *what was in sync* across local files, the OpenAI vector store, and the metadata Excel. The `check-directory-files.py` script was run weekly to catch alignment drift.

### On Technical Choices

**Start simple, add complexity only when forced.** The simple pipeline (v2.0) should have been the starting point. The complex pipeline was built under the assumption that Q&A enrichment was essential — an assumption that proved temporarily correct but ultimately unnecessary.

**Config centralization should happen early.** The migration to `config.py` (Phase 4) resolved a class of bugs that had been persistent: scripts breaking when run from different directories, path inconsistencies between Windows and the format OpenAI expected, etc. Should have done this from the start.

**Streamlit UIs are tempting but usually unnecessary.** The `vector-store-web-ui.py` was satisfying to build but was used maybe twice. The CLI (`vector-store-manager.py`) was faster and more scriptable. Don't build GUIs for tools only you will use.

### On the Dissertation Workflow

**The system worked.** During the most intensive writing period (mid-2025), being able to query "What does Guyer say about the near future?" or "Find passages about algorithmic temporality across Bucher, Bear, and Koselleck" and get back actual text passages with clear provenance was invaluable. It didn't just speed up writing — it enabled a kind of systematic cross-textual analysis that would have been prohibitively tedious to do manually across 300+ sources.

**The investment justified itself.** Building this system took roughly 40 hours across 2024-2025. It saved probably 200+ hours of manual PDF searching during writing. The ratio only makes sense for a project of this scale (300+ sources, 200+ page dissertation), but for that scale, it was transformative.
